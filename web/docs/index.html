Install and build

layerleak currently targets Go 1.24+ and builds directly from source.

go install github.com/brumbelow/layerleak@latest
layerleak --help

Optional local configuration still uses a plain environment file:

cp .env.example .env

Environment

The scanner keeps its runtime surface small: registry endpoints, output location, HTTP timeout, per-file and per-layer caps, repository-wide caps, and optional Postgres persistence. Defaults are applied when a variable is unset. 

export LAYERLEAK_FINDINGS_DIR=findings
export LAYERLEAK_API_ADDR=127.0.0.1:8080
export LAYERLEAK_PERSIST_RAW_SECRETS=0
export LAYERLEAK_TAG_PAGE_SIZE=100
export LAYERLEAK_HTTP_TIMEOUT=30s
export LAYERLEAK_MAX_FILE_BYTES=1048576
export LAYERLEAK_MAX_LAYER_BYTES=536870912
export LAYERLEAK_MAX_LAYER_ENTRIES=50000
export LAYERLEAK_MAX_MANIFEST_BYTES=0
export LAYERLEAK_MAX_CONFIG_BYTES=0
export LAYERLEAK_MAX_TAG_RESPONSE_BYTES=8388608
export LAYERLEAK_MAX_REPOSITORY_TAGS=0
export LAYERLEAK_MAX_REPOSITORY_TARGETS=0
export LAYERLEAK_REGISTRY_REQUEST_ATTEMPTS=2
export LAYERLEAK_DATABASE_URL=postgres://postgres:postgres@localhost:5432/layerleak?sslmode=disable

LAYERLEAK_HTTP_TIMEOUT applies to every registry HTTP call (manifest fetches, blob downloads, tag pages, auth tokens) and accepts any Go duration value. LAYERLEAK_MAX_FILE_BYTES caps decompressed bytes per file inside a layer; files larger than this are classified as oversize and skipped. The size and repository caps suffixed _BYTES, _ENTRIES, _TAGS, and _TARGETS are disabled when set to 0. If enabled and exceeded, the scan fails closed but still writes the partial results produced before the failure.

Result behavior

Actionable findings drive the CLI exit code and stay visible in findings. Likely test, example, or demo placeholders are retained as suppressed findings with disposition, disposition_reason, and line_number metadata for triage. Suppressed findings do not count toward total_findings.

Saved findings files and Postgres persistence default to redacted previews (redacted_value, redacted context_snippet). Raw finding values and raw snippets are written only when LAYERLEAK_PERSIST_RAW_SECRETS=1 is set explicitly. The scan_runs.result_json snapshot stays redacted in either mode. If a configured operational limit trips, layerleak still writes the partial results produced before the failure and exits with status 1.

Secret-safety note

Persistence stores redacted previews by default. Raw findings.value and finding_occurrences.raw_snippet stay empty unless LAYERLEAK_PERSIST_RAW_SECRETS=1. Use a dedicated database or schema for layerleak; for the safest purge path, drop the dedicated database or schema rather than trying to surgically delete individual rows.

Postgres persistence

Migrations are manual by design. The scanner does not auto-create or auto-upgrade the schema. Layerleak requires PostgreSQL server >= 16.13. 

psql "$LAYERLEAK_DATABASE_URL" -f migrations/0001_initial.up.sql
psql "$LAYERLEAK_DATABASE_URL" -f migrations/0002_finding_occurrence_metadata.up.sql
psql "$LAYERLEAK_DATABASE_URL" -f migrations/0003_scan_runs.up.sql

Or apply migrations using the container helper: 

docker run --rm \
  -e LAYERLEAK_DATABASE_URL="$LAYERLEAK_DATABASE_URL" \
  ghcr.io/brumbelow/layerleak:latest \
  layerleak-migrate-up

layerleak-migrate-up is safe to rerun when migrations are already applied; if it detects a partial migration state, it exits non-zero and asks for manual intervention. The schema keeps current deduplicated state with first_seen_at / last_seen_at across repositories, tags, manifests, findings, and occurrences, plus an append-only history in the scan_runs table that stores a redacted snapshot of each scan result.

CLI usage

The primary command surface is still the scanner CLI. Image references can target any public OCI-compliant registry — Docker Hub, GHCR, Quay, GCR, MCR, Amazon ECR Public, or a self-hosted registry — without a Docker daemon. 

./layerleak --help
./layerleak scan --help

./layerleak scan ubuntu
./layerleak scan library/nginx:latest --format json
./layerleak scan alpine:latest --platform linux/amd64
./layerleak scan mongo
./layerleak scan ghcr.io/homebrew/core/hello:latest
./layerleak scan quay.io/prometheus/busybox:latest
./layerleak scan gcr.io/distroless/static:nonroot
./layerleak scan public.ecr.aws/docker/library/alpine:3.20
./layerleak scan mcr.microsoft.com/hello-world:latest

Bare repository names enumerate public tags, resolve each tag to a digest, and group duplicates so the same manifest is only scanned once. If you only want a single image, use an explicit tag (repo:tag) or digest (repo@sha256:...). Ctrl-C cancels the in-flight scan cleanly via context cancellation.

HTTP API

The API is a separate Go binary under cmd/api. It requires LAYERLEAK_DATABASE_URL, serves JSON only, and shuts down gracefully on SIGINT / SIGTERM. 

go run ./cmd/api

Current endpoints:

POST /api/v1/scans
GET  /api/v1/scans/{id}
GET  /api/v1/repositories
GET  /api/v1/repositories/{repository}/scans
GET  /api/v1/repositories/{repository}/findings
GET  /api/v1/findings/{id}

POST /api/v1/scans stays synchronous and returns scan_run_id whenever Postgres persistence is enabled. GET /api/v1/scans/{id} returns the persisted run metadata plus the stored redacted result snapshot. GET /api/v1/repositories/{repository}/scans and .../findings accept an optional ?registry= query parameter (for example ?registry=ghcr.io); when omitted the registry defaults to docker.io. Use it to fetch persisted scans for repositories on GHCR, Quay, GCR, MCR, Amazon ECR Public, or any self-hosted registry.

All API responses mirror the redacted CLI/API view. They never expose raw secret values or raw snippets from Postgres, even when LAYERLEAK_PERSIST_RAW_SECRETS=1 is set on the writer side. The API does not include authentication; for org deployments, keep it on a private network and front it with your own authn/authz gateway.

Browser demo

The GitHub Pages demo is intentionally simulated. It replays a fixed scan command and then shows a fake local Postgres snapshot that matches the storage model conceptually. 

layerleak scan vulnerableHost:latest --platform linux/amd64

The transcript, findings file path, and database rows are all synthetic and versioned in-repo as static fixture data.
